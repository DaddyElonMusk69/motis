## The Operator Contract

Every operator is a folder containing an `operator.py` entry point that exports exactly three things:

```python
STATE: type              # TypedDict defining the state schema
MANIFEST: dict           # Metadata: name, triggers, risk config, node types, agents
def build_graph() -> CompiledStateGraph:  # Returns the executable graph
    ...
```

### Why a flat module?

- The agent generates a self-contained `operator.py` — no class hierarchies
- LangGraph tutorials are module-level — this is the pattern LLMs have seen most
- `exec()` in the runtime → extract the 3 exports → done
- The existing `OperatorBase` class stays for hand-coded developer operators

Both paths produce the same thing: a compiled graph + state + manifest.

### The MANIFEST `agents` section (optional)

For operators that need multi-step agent reasoning (e.g., research teams, trading desks),
MANIFEST can declare named agents. Each agent gets its own system prompt, tool whitelist,
skill whitelist, and iteration limit:

```python
MANIFEST = {
    "name": "Crypto Trading Desk",
    "type": "research",
    "agents": {
        "funding_analyst": {
            "role": "Funding Rate & Basis Analyst",
            "system_prompt": "You are a senior derivatives analyst...",
            "tools": ["bash", "read_file", "load_skill"],
            "skills": ["perp-funding-basis", "okx-market"],
            "max_iterations": 50,
            "model_name": None,  # None = use user's default model
        },
        "risk_manager": {
            "role": "Desk Risk Manager",
            "system_prompt": "You are the head risk manager...",
            "tools": ["bash", "read_file", "write_file", "load_skill"],
            "skills": ["risk-analysis", "asset-allocation"],
        },
    },
    "nodes": [
        {"name": "fetch_meta",       "type": "DATA"},
        {"name": "funding_analysis", "type": "REASON", "agent": "funding_analyst"},
        {"name": "risk_synthesis",   "type": "REASON", "agent": "risk_manager"},
    ],
}
```

When a REASON node references an `"agent"` key, it spawns a **new, scoped agent instance**
(not the master agent). That agent runs a full ReAct loop with the specified tools and skills.

## The 5 Node Types

Every node in the graph is one of these. The master agent classifies each step
during decomposition:

| Type | Nature | What it does | Who owns risk? |
|---|---|---|---|
| **DATA** | I/O | Fetch OHLCV, orderbook, funding rates, news | N/A |
| **COMPUTE** | Pure function | Indicators, patterns, position sizing, SL/TP calc | N/A |
| **REASON** | LLM call or agent loop | Interpret data, decide entry/exit, deep research | Prompt author |
| **GUARD** | Deterministic | Position limits, daily loss check, leverage cap, cooldown | **The operator itself** |
| **EXECUTE** | I/O | Submit orders, set SL/TP, cancel orders | MCP layer |

#### REASON nodes: two modes

- **Single-call** — `reason_call(prompt, ...)` for quick structured decisions ("long or flat?")
- **Agent-loop** — `run_agent(agent_id, context)` for multi-step work (fetch data, analyze,
  write report). References an agent declared in `MANIFEST["agents"]`. Each spawned agent
  gets its own scoped tool/skill whitelist and runs independently of the master agent.

### The critical difference from v1

> **GUARD nodes are generated by the master agent as part of the operator code.**
> The platform does not inject or modify them. If the agent builds a bad operator
> without proper guards, the Quality Gate catches it before deployment.

## The Quality Gate

This is the most important part of the design. Since the platform doesn't enforce safety,
the master agent must verify these checks before an operator can graduate from `draft` to
`paper` or `live`:

### The 10-Point Checklist

```python
QUALITY_CHECKS = {
    # ── Hard Safety (must ALL pass) ─────────────────────────────────────
    "hard_stop_loss": {
        "description": "Every EXECUTE node must set a stop-loss on every order",
        "severity": "blocker",
        "check": "AST scan: every submit_order() call includes sl= parameter",
    },
    "position_sizing_cap": {
        "description": "No single position can exceed the risk config's max_position_size_pct",
        "severity": "blocker",
        "check": "AST scan: size calculation includes min() or cap logic",
    },
    "daily_loss_killswitch": {
        "description": "Graph must check cumulative daily P&L and halt if breached",
        "severity": "blocker",
        "check": "GUARD node exists that reads daily_pnl and returns should_exit=True",
    },
    "leverage_cap": {
        "description": "Leverage never exceeds MANIFEST.risk.max_leverage",
        "severity": "blocker",
        "check": "COMPUTE or GUARD node enforces leverage limit",
    },
    "error_handling": {
        "description": "DATA nodes handle API failures gracefully (retry or exit, not crash)",
        "severity": "blocker",
        "check": "try/except around all call_skill() and external calls in DATA nodes",
    },

    # ── Structural Quality (should pass) ────────────────────────────────
    "state_completeness": {
        "description": "Every field in STATE is written by at least one node",
        "severity": "warning",
        "check": "Static analysis of return dicts vs STATE fields",
    },
    "guard_before_execute": {
        "description": "A GUARD node runs between every REASON and EXECUTE node",
        "severity": "warning",
        "check": "Graph topology analysis",
    },
    "logging": {
        "description": "Every node writes at least one log_event() for observability",
        "severity": "warning",
        "check": "AST scan for log_event() calls in each node function",
    },

    # ── Performance Quality (should pass before live, not before paper) ──
    "backtest_sharpe": {
        "description": "Backtest Sharpe ratio > 0.5 over the test period",
        "severity": "live_gate",
        "check": "Run backtest, check metrics.sharpe",
    },
    "backtest_max_drawdown": {
        "description": "Backtest max drawdown < 2x the daily loss killswitch threshold",
        "severity": "live_gate",
        "check": "Run backtest, check metrics.max_drawdown_pct",
    },
}
```

### Gate transitions

```
                          Quality Gate (5 hard checks)
DRAFT ──────────────────────────────────────────────────→ PAPER
                                                           │
                          Quality Gate (hard + structural   │
                            + performance backtests)        │
                                                           ▼
                                                         LIVE
```

- **draft → paper**: All 5 `blocker` checks must pass. Structural `warning` checks are surfaced
  to the user but don't block.
- **paper → live**: All `blocker` + all `warning` + both `live_gate` checks must pass.
  The user must also explicitly confirm ("Yes, connect my exchange and go live").

### Who runs the Quality Gate?

The **master agent** runs it using the `terminal` tool. It:

1. Loads the operator's `graph_code` 
2. Parses the AST
3. Checks each item in the checklist
4. Reports pass/fail to the user
5. If failures, suggests specific fixes
6. Re-generates the code and re-checks

This is why the operator-builder SKILL.md includes the full checklist — the agent knows
what it needs to achieve before the operator is considered complete.

## The Operator Lifecycle

```
                    ┌─────────────────────────────────────────┐
                    │            OPERATOR STATES               │
                    ├───────────┬──────────────────────────────┤
                    │ DRAFT     │ Code generated, not validated │
                    │ VALIDATING│ Quality Gate running           │
                    │ PAPER     │ Running on schedule, no $      │
                    │ LIVE      │ Running with real money         │
                    │ PAUSED    │ Halted (manual or kill-switch) │
                    │ ARCHIVED  │ Deactivated permanently        │
                    └───────────┴──────────────────────────────┘

    ┌───────┐   generate   ┌──────────┐  pass gate  ┌───────┐
    │ DRAFT │─────────────→│VALIDATING│────────────→│ PAPER │
    └───┬───┘              └────┬─────┘             └───┬───┘
        │                      │ fail                   │
        │◄─────────────────────┘ (back to draft         │ pass gate
        │                        with fix suggestions)  │ + user confirm
        │                                               ▼
        │                                           ┌───────┐
        │                                           │ LIVE  │
        │                                           └───┬───┘
        │                                               │
        │                                    kill-switch │ or manual
        │                                               ▼
        │                                           ┌────────┐
        │                                           │ PAUSED │
        │                                           └───┬────┘
        │                                               │
        └───────────────────────────────────────────────┘
                        iterate / fix
```

## How the Master Agent Builds an Operator

### Step 1: Load the operator-builder skill

The agent loads `skills/operator-builder/SKILL.md` which contains:

- The module contract (STATE + MANIFEST + build_graph)
- The 5 node types and when to use each
- The Quality Gate checklist (all 10 checks)
- 3 complete annotated examples
- The SDK surface (what functions are available)
- Anti-patterns ("never skip stop-loss", "never put risk logic in REASON nodes")

### Step 2: Decompose intent

```
User: "Open a BTC long when the 4h shows bullish BOS after a liquidity sweep,
       with 2% risk per trade and a hard stop below the sweep low"

Agent decomposes:
  1. Fetch BTC/USDT 4h OHLCV (300 candles)          → DATA
  2. Detect swing highs/lows + BOS + CHoCH           → COMPUTE (smc skill)
  3. Detect liquidity sweeps                          → COMPUTE (smc skill)
  4. LLM decides: is this a valid long setup?         → REASON
  5. Calculate position size for 2% risk              → COMPUTE (deterministic)
  6. Calculate SL below sweep low, TP at next OB      → COMPUTE (deterministic)
  7. Check: daily loss not breached                   → GUARD
  8. Check: position count within limits               → GUARD
  9. Check: leverage within limits                     → GUARD
  10. Submit order with SL/TP attached                 → EXECUTE
```

### Step 3: Generate code

The agent generates the full Python module (see example below).

### Step 4: Run Quality Gate

```python
# The agent runs this via the terminal tool:
from motis_operator.validation import run_quality_gate

results = run_quality_gate(graph_code)
# → {"hard_stop_loss": PASS, "daily_loss_killswitch": PASS, ...}
```

### Step 5: Iterate if needed

If checks fail, the agent explains what failed and fixes the code. It re-runs
the gate until all checks pass.

### Step 6: Deploy

```python
operator_create(
    name="BTC SMC Long",
    type="paper_trade",
    spec={"graph_code": graph_code_string}
)
```
